# Schema Reference

Complete field-by-field reference for the OpenCode Skill Specification.

---

## Required Fields

### `name`

**Purpose:** Machine-friendly identifier for the skill.

**Format:**
- Lowercase letters, numbers, and hyphens only
- Must start with letter or number
- Maximum 50 characters
- No spaces or underscores

**Examples:**
```yaml
name: git-workflow          # ✅ Good
name: code-review          # ✅ Good
name: Git Workflow         # ❌ Bad - uppercase and spaces
name: code_review          # ❌ Bad - underscore
```

**Usage:**
- File naming
- Registry lookups
- Cross-references between skills

---

### `description`

**Purpose:** Human-readable trigger phrase that tells users when to use this skill.

**Format:**
- Recommended: Start with "Use when..."
- 10-30 words ideal
- Action-oriented
- Specific enough to trigger recall

**Examples:**
```yaml
description: Use when planning safe git commits and repository cleanup
description: Use when reviewing implementation quality before merge
description: Use when synthesizing research into decision-ready reports
```

**Anti-patterns:**
```yaml
description: A skill for git stuff                    # ❌ Too vague
description: This skill helps you do git things        # ❌ Passive
description: Git workflow management system            # ❌ Not actionable
```

---

### `type`

**Purpose:** Defines the instructional mode and interaction pattern.

**Allowed Values:**

| Type | Description | Best For |
|------|-------------|----------|
| `technique` | Step-by-step method with clear procedure | Concrete, repeatable processes |
| `pattern` | Mental model or way of thinking | Conceptual frameworks, decision guides |
| `reference` | API docs, syntax guides, lookup tables | Documentation-heavy content |
| `discipline` | Rules or requirements to follow | Standards enforcement, checklists |

**Examples:**
```yaml
type: technique    # "Do X, then Y, then Z"
type: pattern      # "Think about it this way..."
type: reference    # "Here's the syntax for..."
type: discipline   # "You MUST always..."
```

---

### `category`

**Purpose:** Primary management bucket for organizing skills.

**Recommended Values:**

| Category | Use For |
|----------|---------|
| `engineering` | General software engineering workflows |
| `frontend` | UI, UX, browser work, styling |
| `backend` | APIs, services, server logic |
| `devops` | Infrastructure, deployment, operations |
| `testing` | Unit, integration, E2E testing |
| `security` | Security reviews, hardening |
| `research` | Investigation, discovery, analysis |
| `writing` | Documentation, communication |
| `workflow` | Process, collaboration, productivity |
| `product` | Product thinking, strategy |
| `general` | Cross-functional or uncategorized |

**Examples:**
```yaml
category: engineering    # Git workflow, code review
category: frontend       # UI polish, accessibility
category: security       # Threat modeling, input validation
```

**Note:** Choose the single best fit. Don't multi-select.

---

### `tags`

**Purpose:** Searchable keywords for discovery and filtering.

**Characteristics:**
- Narrow, specific terms
- Concrete nouns and verbs
- Think "what would someone search for?"
- 3-5 tags recommended
- AI-generated by default, user-confirmed

**Examples:**
```yaml
tags:
  - git
  - commits
  - repository
  - history
  - cleanup
```

**Anti-patterns:**
```yaml
tags:
  - code          # ❌ Too broad
tags:
  - improving     # ❌ Process word, not keyword
tags:
  - validation    # ❌ Vague without context
```

**See also:** [Tags vs Topics Guide](tags-topics.md)

---

### `topics`

**Purpose:** Thematic groups for browsing and organization.

**Characteristics:**
- Broader than tags
- Think "what theme does this belong to?"
- 1-3 topics recommended
- AI-generated by default, user-confirmed

**Examples:**
```yaml
topics:
  - version-control
  - code-hygiene
```

**Anti-patterns:**
```yaml
topics:
  - software-engineering    # ❌ Too broad (every engineering skill)
  - execution-patterns      # ❌ Too generic
```

**See also:** [Tags vs Topics Guide](tags-topics.md)

---

### `boundary`

**Purpose:** One-sentence scope definition that sets expectations.

**Format:**
- Single sentence
- Starts with "Focused on..."
- Concrete, not aspirational
- Specific enough to judge fit

**Good Examples:**
```yaml
boundary: "Focused on local git hygiene, commit planning, and safe history editing"
boundary: "Covers input validation patterns and common injection prevention"
boundary: "Addresses frontend accessibility checks for WCAG 2.1 AA compliance"
```

**Bad Examples:**
```yaml
boundary: "Helps with coding tasks"                          # ❌ Way too vague
boundary: "Makes you a better developer"                     # ❌ Aspirational fluff
boundary: "Everything about git"                             # ❌ Too broad
```

**Why this matters:**
The boundary is your contract with users. It prevents:
- Wrong expectations
- Misuse for out-of-scope problems
- Feature creep over time

---

### `maturity`

**Purpose:** Lifecycle stage that signals trust and readiness.

**Allowed Values:**

| Stage | Meaning | When to Use |
|-------|---------|-------------|
| `draft` | Early idea, needs shaping | Just created, structure uncertain |
| `alpha` | Experimental but usable | Core there, needs testing |
| `beta` | Tested, mostly stable | Works well, gathering feedback |
| `stable` | Recommended for regular use | Battle-tested, maintained |
| `deprecated` | Kept for reference only | Replaced by better alternative |

**Examples:**
```yaml
maturity: draft      # "I just wrote this, it might not work"
maturity: alpha      # "Try it out, but expect issues"
maturity: beta       # "Good enough for most uses"
maturity: stable     # "Production-ready, recommended"
maturity: deprecated # "Don't use for new work"
```

**Why stages over scores:**
- **Stages are actionable** — "This is beta" tells you something concrete
- **Scores are meaningless** — What does "8.5/10" mean?
- **Stages prevent gaming** — Harder to manipulate than numbers
- **Stages map to process** — Natural review gates at each transition

## Optional Fields

### `non_goals`

**Purpose:** Explicit exclusions to prevent misuse and scope creep.

**Format:**
- List of strings
- Each states something the skill explicitly does NOT do
- 1-2 items typical, 3+ if scope is tricky

**Examples:**
```yaml
non_goals:
  - "Teaching Git fundamentals"
  - "Managing remote repository policy"
  - "Automating branch protection rules"
```

**Why include non-goals:**
- Prevents "why doesn't this do X?" confusion
- Makes boundary enforcement explicit
- Helps authors resist scope creep
- Guides users to the right tool

**Comparison:**
- `boundary` says "here's what I do"
- `non_goals` says "here's what I DON'T do, on purpose"

## Complete Examples

### Example 1: Git Workflow (Technique)
```yaml
---
name: git-workflow
description: Use when planning safe git commits and repository cleanup
type: technique
category: engineering
tags:
  - git
  - commits
  - repository
  - history
  - cleanup
topics:
  - version-control
  - code-hygiene
boundary: "Focused on local git hygiene, commit planning, and safe history editing"
maturity: beta
non_goals:
  - "Teaching Git fundamentals"
  - "Managing remote branch policies"
---
```

### Example 2: Code Review (Pattern)
```yaml
---
name: code-review
description: Use when reviewing implementation quality before merge
type: pattern
category: engineering
tags:
  - code-review
  - quality
  - maintainability
  - risk
topics:
  - quality-engineering
  - code-review
boundary: "Covers structured code review, risk identification, and improvement guidance"
maturity: beta
non_goals:
  - "Automated linting or static analysis"
  - "Security penetration testing"
---
```

### Example 3: API Reference (Reference)
```yaml
---
name: postgres-quick-ref
description: Use when writing PostgreSQL queries and need syntax reminders
type: reference
category: backend
tags:
  - postgres
  - sql
  - database
  - queries
topics:
  - databases
  - sql-reference
boundary: "Provides common PostgreSQL syntax, functions, and query patterns"
maturity: stable
---
```

## Validation Rules

### Automated Checks

A validator SHOULD check:

1. **name**
   - Matches `^[a-z0-9]+(-[a-z0-9]+)*$`
   - ≤ 50 characters

2. **description**
   - Not empty
   - ≤ 200 characters recommended

3. **type**
   - One of: `technique`, `pattern`, `reference`, `discipline`

4. **category**
   - One of recommended values (or custom with warning)

5. **tags**
   - At least 1 tag
   - Each tag: lowercase, kebab-case, 3-30 characters

6. **topics**
   - At least 1 topic
   - Each topic: kebab-case, meaningful (not generic)

7. **boundary**
   - Not empty
   - Starts with "Focused on" or similar scope phrase
   - ≤ 200 characters

8. **maturity**
   - One of: `draft`, `alpha`, `beta`, `stable`, `deprecated`

### Warnings (Not Errors)

A validator MAY warn about:

- Tags that are too generic (`code`, `software`, `development`)
- Topics that are too broad (`software-engineering`)
- Boundary that's vague or aspirational
- Missing `non_goals` when `type` is `discipline`
- Description that doesn't start with "Use when"

## Migration from Minimal Schema

If you have skills with only `name` and `description`:

1. **Add required fields**:
   - `type`: Infer from content (default to `technique`)
   - `category`: Choose best fit (default to `general`)
   - `boundary`: Extract one-sentence scope from content
   - `maturity`: Start everything as `draft`
   - `tags`, `topics`: Generate with AI/tooling, review manually

2. **Gradual improvement**:
   - Promote `maturity` as skills get tested
   - Refine `tags/topics` based on usage patterns
   - Add `non_goals` when misuse is observed

3. **Tooling**:
   - Use generators that auto-suggest missing fields
   - Use linters to catch low-quality metadata
   - Use catalogs to browse and identify gaps
