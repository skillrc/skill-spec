# Contributing

Thanks for helping improve the SKILL specification.

## Contribution Principles

- Keep the spec lightweight and practical.
- Prefer management and clarity improvements over early scoring complexity.
- Preserve a clear distinction between required schema fields and authoring experience.
- When proposing new fields, explain the governance value they unlock.

## How to Contribute

1. Open an issue or draft proposal describing the problem.
2. Explain why the current schema is insufficient.
3. Propose the smallest useful change.
4. Include examples showing how the change improves real skill authoring.

## Documentation Expectations

- Use concise language.
- Prefer examples over abstract theory.
- Keep terminology consistent across README, SPEC, and examples.
- If a field is required in the schema but generated by tooling, state that explicitly.

## Pull Request Checklist

- Update `SPEC.md` if the schema changes.
- Update examples when authoring behavior changes.
- Update `CHANGELOG.md` with a summary.
- Explain compatibility implications if existing skill files would be affected.
